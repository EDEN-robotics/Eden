Working Cognitive Layer

Short-term memory (Redis), long-term relational memory (Postgres + Vector DB), and personality module.

ROS MCP Action Layer

Simulation

Gazebo World with
Semantic Mapping

Custom apartment in Gazebo Fortress — two rooms populated with objects, each labeled in a YAML semantic map so the planning layer can reason about what's where.

Semantic Map Structure

world → rooms → furniture → items

Each object entry has: id (label), pose (XYZ), type (category), sdf_models (link to sim), description

Navigation Hints

Pre-computed safe approach positions per fixture. Robot knows where to stand to reach an object without collision.

dining_table:
  south: (0.0, -1.5)
  north: (0.0, 1.5)
  east: (1.5, 0.0)
SDF Model Resolution

When Gazebo reports a sensor hit on teacup1_body, the planner resolves it via YAML to logical object teacup_white on dining_table in dining_room.

Querying the Semantic Map
import yaml
with open('apartment_semantic_map.yaml') as f:
  world = yaml.safe_load(f)

Architecture

Offloaded Compute:
How It Works

Each robot is fully independent. The laptop is just a GPU server — it hosts the compute, not the decisions.

Robot Side (ESP32)

ESP-CAM32 streams video frames over WiFi. IMU sends orientation data. Receives motor commands back. That's it — just a sensor/actuator node.

Laptop Side

Runs 3 completely independent EDEN pipelines. Each robot gets its own full Input → Context → Cognitive → Planning → Action stack. No shared decisions between robots.

Communication

micro-ROS on ESP32 → ROS 2 DDS on laptop over WiFi. Each robot in its own namespace (/eden_1/, /eden_2/, /eden_3/). ~5-10ms local network latency.
